# 微压缩 (Micro-compaction) 实施指南

本指南规定了如何管理大型工具输出，以防止 Gemini CLI 会话因上下文过载而导致的性能下降、幻觉增加或崩溃。

## 场景识别

当执行以下工具时，应根据输出规模调整策略：
- `run_shell_command`: 执行构建、测试或全量搜索时。
- `grep_search`: 在大型仓库中进行模糊搜索时。
- `read_file`: 读取非常大的日志或数据文件时。

## 核心策略：本地持久化 + 智能摘要

### 1. 自动重定向 (Redirection)
如果你预计输出可能很大（例如 `npm test` 或 `grep -r`），请不要直接运行，应将其重定向到临时文件：
```powershell
# 推荐做法
npm test > temp_test_results.log 2>&1
```
然后，使用 `read_file` 的 `start_line` 和 `end_line` 参数来探测文件内容。

### 2. 生成“技术快照” (Technical Snapshot)
不要将原始输出喂给 LLM。你需要生成一个摘要，包含：
- **统计信息**：总行数、成功/失败的数量。
- **错误上下文**：提取关键的 Stack Trace 或报错行。
- **关键结构**：对于搜索任务，仅列出受影响的文件路径及每个文件中的匹配行数。

### 3. 分块阅读 (Paging)
如果必须阅读完整输出，应在多个 Turn 中分块读取（例如每次 100 行），并由你自己记录中间状态。

## 角色指引

你作为 Agent，在执行此类操作时，应像人类工程师一样进行“有损压缩”。
- **丢弃**：重复的进度条信息、常规的编译日志、无关的警告。
- **保留**：具体的异常类名、出错的文件行号、关键的性能指标。

---
*通过此策略，可使 Gemini CLI 的有效上下文窗口提升 10 倍以上。*
